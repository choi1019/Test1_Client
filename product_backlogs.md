# Product Backlog: Video Streaming Client

## 제품 목표

서버로부터 비디오 스트리밍을 수신하여 클라이언트 화면에 안정적으로 표시하는 데스크톱 애플리케이션을 개발한다. 초기 버전은 연결, 수신, 디코딩, 표시, 오류 처리에 집중하고 이후 성능 측정, 사용자 설정, 재연결, 테스트 자동화를 확장한다.

## 대상 사용자

- 스트리밍 서버의 실시간 영상을 확인해야 하는 운영자
- 영상 송수신 기능을 검증해야 하는 개발자 또는 QA 담당자
- 네트워크 상태와 영상 수신 품질을 확인해야 하는 테스트 담당자

## 주요 가정

- 클라이언트는 C++ 기반으로 개발한다.
- 화면 표시는 Qt 위젯 또는 Qt Quick 중 하나를 사용한다.
- 영상 디코딩 및 프레임 처리는 OpenCV를 우선 활용한다.
- 서버 스트리밍 프로토콜은 MVP 단계에서 확정하거나 설정 가능하도록 설계한다.
- 초기 지원 프로토콜 후보는 RTSP, HTTP MJPEG, WebSocket 프레임 스트림 중 하나이다.

## 우선순위 기준

- P0: MVP 구현에 반드시 필요한 항목
- P1: 실사용 안정성과 품질 확보에 필요한 항목
- P2: 사용성, 운영 편의, 확장성 향상을 위한 항목
- P3: 장기 개선 또는 선택 기능

## Definition of Done

- 기능 요구사항의 Acceptance Criteria를 모두 만족한다.
- CMake 빌드가 성공한다.
- 신규 또는 변경 로직에 대한 테스트가 추가된다.
- UI가 스트리밍 중에도 응답성을 유지한다.
- 오류 발생 시 프로그램이 비정상 종료되지 않는다.
- 사용자 설정과 로그가 의도한 위치에 저장된다.
- README 또는 별도 문서에 실행 방법과 테스트 방법이 기록된다.

## 주요 리스크

- 서버 프로토콜이 확정되지 않으면 수신 모듈 구현 범위가 흔들릴 수 있다.
- 고해상도 또는 고FPS 스트림에서 디코딩과 UI 갱신 성능 문제가 발생할 수 있다.
- 네트워크 지연과 패킷 손실 상황에서 버퍼링 정책을 잘못 선택하면 실시간성이 저하될 수 있다.
- GUI 스레드와 수신 스레드 간 동기화가 부정확하면 충돌 또는 화면 끊김이 발생할 수 있다.
